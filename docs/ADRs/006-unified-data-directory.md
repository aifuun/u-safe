# ADR-006: 统一数据目录架构

**状态**: 已接受
**日期**: 2026-03-22
**决策者**: 开发团队
**标签**: 架构, 数据存储, 安全

## 背景

在 Issue #43 中发现，应用的数据存储存在架构缺陷：

### 问题现状

不同模块使用不同的数据目录：

```
数据库模块  → ./.u-safe/u-safe.db          (项目目录)
密码模块    → ~/Library/Application Support/.u-safe/keys/password.hash  (系统目录)
密钥模块    → 相对路径 .u-safe/keys/master.key  (不确定)
```

### 问题影响

1. **违背 PRD 设计意图**：PRD 明确要求所有数据存储在 U 盘的 `.u-safe/` 目录
2. **便携性问题**：密码存储在系统目录，无法实现 U 盘便携
3. **数据管理混乱**：用户数据分散在多个位置，难以备份和迁移
4. **阻塞功能开发**：日志系统不知道应该使用哪个目录

## 决策

**所有应用数据统一存储在 `.u-safe/` 目录下。**

### 目录结构

```
.u-safe/                    # 统一数据目录（当前目录或 U 盘根目录）
├── u-safe.db              # SQLite 数据库
├── keys/                  # 密钥存储
│   ├── password.hash      # 密码哈希（Argon2id）
│   └── master.key         # 加密的主密钥
├── data/                  # 加密文件数据（未来）
└── logs/                  # 应用日志（未来）
```

### 实现方式

1. **使用 `usb_detection::get_data_dir()`**
   - 所有模块统一调用此函数获取数据目录路径
   - MVP 阶段返回 `./.u-safe`（项目根目录）
   - 未来支持 U 盘检测后，自动切换到 U 盘路径

2. **数据迁移逻辑**
   - 应用启动时检测旧数据（系统目录）
   - 自动迁移到新位置（.u-safe/）
   - 保留旧数据不删除（用户手动清理）

3. **修改范围**
   - `password.rs`: `get_hash_file_path()` 使用 `usb_detection::get_data_dir()`
   - `keystore.rs`: `get_master_key_path()` 使用 `usb_detection::get_data_dir()`
   - `lib.rs`: 添加 `migrate_old_data()` 函数

## 替代方案

### 方案 A：混合模式（已拒绝）

**描述**：敏感数据（密码哈希）存储在系统目录，其他数据存储在 .u-safe/

**优点**：
- 符合操作系统安全最佳实践
- 防止密码随 U 盘物理丢失而泄露

**缺点**：
- 违背产品设计意图（U 盘便携）
- 用户无法在多台电脑上使用同一个 U 盘
- 数据管理复杂度增加

**拒绝理由**：产品定位是 U 盘加密工具，便携性是核心需求。

### 方案 B：完全系统目录（已拒绝）

**描述**：所有数据存储在 `~/Library/Application Support/.u-safe/`

**优点**：
- 符合操作系统惯例
- 用户权限管理简单

**缺点**：
- 完全违背 PRD 设计
- 无法实现 U 盘便携
- 失去产品核心竞争力

**拒绝理由**：与产品定位冲突。

## 安全性考虑

### 密码安全

**问题**：密码哈希存储在 U 盘，如果 U 盘物理丢失，攻击者可以暴力破解。

**缓解措施**：

1. **强密码策略**
   - 最小长度 8 字符（Issue #36 调整后）
   - 提示用户设置复杂密码

2. **Argon2id 参数**
   ```rust
   Argon2id {
       m_cost: 65536,     // 64 MB 内存
       t_cost: 3,         // 3 次迭代
       p_cost: 4,         // 4 个并行线程
   }
   ```
   - 暴力破解成本极高（单次尝试 ~100ms）
   - 攻击者需要数年时间破解强密码

3. **用户教育**
   - 文档明确说明风险
   - 建议启用 U 盘硬件加密（如 BitLocker）

### 数据完整性

- 使用 AES-256-GCM 确保加密数据未被篡改
- 主密钥用密码派生密钥加密存储
- 所有敏感操作记录日志

## 跨平台兼容性

### macOS
- 数据目录：`./u-safe/` 或 `/Volumes/USB_NAME/.u-safe/`
- 权限：应用自动创建目录，无需特殊权限

### Windows
- 数据目录：`.\.u-safe\` 或 `E:\.u-safe\`
- 权限：同上

### Linux（未来支持）
- 数据目录：`./.u-safe/` 或 `/media/USER/USB_NAME/.u-safe/`
- 权限：需要挂载点读写权限

## 后果

### 正面影响

1. **架构统一**：所有模块使用相同的数据目录逻辑
2. **便携性实现**：满足 PRD 设计意图
3. **功能解锁**：日志系统可以正常开发
4. **用户体验**：数据集中管理，易于备份

### 负面影响

1. **迁移成本**：需要实现数据迁移逻辑
2. **安全性权衡**：密码哈希暴露在 U 盘（可接受风险）
3. **测试复杂度**：需要测试迁移逻辑的正确性

### 风险缓解

- 数据迁移自动执行，保留旧数据
- 充分的单元测试和集成测试
- 日志记录迁移过程

## 实施计划

### Phase 1: 修复密码管理模块 ✅
- 修改 `password.rs::get_hash_file_path()`
- 使用 `usb_detection::get_data_dir()`

### Phase 2: 修复密钥存储模块 ✅
- 修改 `keystore.rs`
- 移除硬编码路径
- 添加 `get_master_key_path()`

### Phase 3: 添加数据迁移逻辑 ✅
- 在 `lib.rs` 添加 `migrate_old_data()`
- 应用启动时自动迁移
- 日志记录迁移过程

### Phase 4: 测试验证 ✅
- 单元测试：路径函数返回正确
- 集成测试：文件存储位置正确
- 手动测试：迁移逻辑工作正常

### Phase 5: 文档更新 🚧
- 创建本 ADR
- 更新 PRD 目录结构说明
- 更新 README 用户指南
