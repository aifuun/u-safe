Skill Creator（skill-creator）详细解释说明

本页面是对 skills/skill-creator/SKILL.md 的中文化、结构化说明。目标是帮助你快速掌握该技能在「创建 skill、验证效果、迭代优化、提升触发率」四个方面的完整工作流。

适用对象：想构建或优化 Skill 的开发者 核心理念：先小样本验证，再多轮迭代 关键产物：SKILL.md + eval 数据 + 用户反馈

1. 这个 Skill 是做什么的？

skill-creator 是一个“技能开发教练”。它不是只帮你写一个 SKILL.md，而是带你走完从想法到可用技能的完整闭环：

  1. 明确意图和触发条件（这个 skill 到底什么时候该被调用）
  2. 写出初版 skill
  3. 设计测试用例并运行（含基线对比）
  4. 结合定量评测 + 用户主观反馈做改进
  5. 必要时优化 description，提升触发准确率
一句话理解：它是“以评测驱动的 skill 研发流程”，而不是一次性写文档。

2. 核心流程（最重要）

阶段 A：定义与访谈

  • 技能要解决什么问题？
  • 什么用户表达会触发它？
  • 输出格式是什么？
  • 是否需要测试用例（客观任务优先）？

阶段 B：写初版 SKILL

  • 完善 frontmatter：name、description
  • description 要写清“做什么 + 何时触发”
  • 正文描述流程、边界、示例与资源结构

阶段 C：跑评测与对比

  • 每个用例同时跑 with-skill 与 baseline
  • 记录耗时/Token，生成 grading.json
  • 聚合 benchmark，启动 viewer 给用户评审

阶段 D：迭代优化

  • 读 feedback.json，定位问题类型
  • 避免“为某个样例过拟合”
  • 继续下一轮迭代直到用户满意

3. 如何写出更好的 SKILL.md

要点说明实践建议
触发描述（description） 是技能是否被调用的主要依据 写清任务场景 + 同义表达 + 边界；略“积极触发”
渐进加载 metadata 常驻，正文触发后加载，resources 按需读取 正文保持清晰，复杂内容放 references/ 或 scripts/
结构组织 多领域技能应按变体拆分（例如按云厂商） 用目录层级减少无关上下文注入
写作风格 少堆硬性命令，多解释“为什么” 让模型理解目标与原因，提升泛化能力

4. 评测执行细则（实战）

4.1 用例与目录规范

  • 测试用例写入 evals/evals.json（先写 prompt，不急着写断言）
  • 结果落盘到 <skill-name>-workspace/iteration-N/eval-*/
  • 每个 eval 保存 eval_metadata.json（含名称、prompt、assertions）

4.2 同步启动对比运行

  • 每条测试同时跑两组：with_skill 与 baseline
  • 创建新 skill 时 baseline 通常为 without_skill
  • 优化旧 skill 时 baseline 可为“旧快照版本”

4.3 中途不要空等：并行起草断言

  • 运行过程中补充可客观验证的 assertions
  • 主观质量（文风、美感）优先让人评审，不要强行量化

4.4 完成后聚合与展示

python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
nohup python eval-viewer/generate_review.py <workspace>/iteration-N --benchmark <workspace>/iteration-N/benchmark.json &

viewer 会同时展示输出样例与 benchmark，用户评审后会生成 feedback.json。

4.5 必须抓取 timing

每个运行完成时立刻记录 total_tokens 和 duration_ms 到 timing.json，这是唯一稳定来源。

5. 迭代改进的原则

  • 泛化优先：从反馈中抽象共性，不要只修单个样例。
  • 提示精简：删掉低价值指令，避免模型做无效劳动。
  • 解释原因：告诉模型“为什么这样做”，比生硬约束更稳。
  • 沉淀重复动作：若多个用例重复写同类脚本，提炼到 scripts/。
终止条件：用户明确满意、反馈为空、或继续迭代已无明显收益。

6. Description 优化（触发率优化）

  1. 构造约 20 条触发评测 query（应触发 + 不应触发）
  2. 使用 assets/eval_review.html 让用户审阅并导出 eval 集
  3. 后台运行优化循环（最多 5 轮）
  4. 用 best_description 回写 SKILL frontmatter
python -m scripts.run_loop \
  --eval-set <trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <当前会话模型ID> \
  --max-iterations 5 \
  --verbose

关键点：评测 query 要“真实、具体、带上下文”，避免过于简短的玩具样例。

7. 不同环境下怎么落地

环境可用能力建议做法
Claude Code 有子代理、可并行、可打开 viewer 按完整流程执行（并行评测 + benchmark + viewer）
Claude.ai 无子代理、评测并行能力弱 顺序跑用例，重点做人工评审，弱化基线 benchmark
Cowork/无显示环境 可运行脚本但可能无浏览器界面 viewer 用 --static 输出独立 HTML；反馈用下载文件回收

8. 常见误区与规避

  • 误区：只写好看提示词，不做评测。
    规避：至少 2-3 条真实用例 + 一轮用户反馈。
  • 误区：断言字段随意命名。
    规避：grading.json 必须使用 text/passed/evidence。
  • 误区：先自我主观评价，再给用户看。
    规避：先生成 viewer，让人类尽早参与审阅。
  • 误区：description 只写“功能”，不写“何时触发”。
    规避：同时覆盖任务类型、上下文线索、边界条件。

9. 一页版执行清单（可直接照做）

  1. 澄清目标、触发场景、输出格式
  2. 写初版 SKILL.md
  3. 写 2-3 条 eval prompt 到 evals/evals.json
  4. 并行跑 with-skill 与 baseline
  5. 运行中补 assertions，完成后做 grading + benchmark
  6. 用 generate_review.py 让用户评审，读取 feedback.json
  7. 根据反馈改进 skill，进入下一迭代
  8. 完成后做 description 优化并回写

如果你只想先最小可用：完成前 1~6 步即可形成有效闭环。
